
## 概述
项目采用了清晰的分层架构来组织数据库相关代码，遵循关注点分离原则，将数据访问、业务逻辑和表现层有效分离。整个数据库层采用 Drizzle ORM 作为核心，配合 PostgreSQL 数据库。

## 技术栈

- **ORM**: Drizzle ORM with TypeScript
- **数据库**: PostgreSQL (Neon Serverless)
- **连接方式**: neon-http (支持 Edge Runtime 和 Cloudflare Workers)
- **类型安全**: 完整的 TypeScript 类型推断

## 架构分层

### 1. 数据库连接层 (`src/server/db/index.ts`)

```typescript
// 使用 neon-http 连接，支持 Edge Runtime 和 Cloudflare Workers
const sql = neon(env.DATABASE_URL);
const db = drizzle(sql, { schema });
```

**职责**:
- 建立数据库连接
- 配置 Drizzle ORM 实例
- 导出全局数据库实例

### 2. Schema 定义层 (`src/server/db/schema.ts`)

定义了完整的数据库表结构，包括：

- **用户相关表**:
  - `user`: 用户基本信息
  - `session`: 用户会话管理
  - `account`: 第三方账户关联
  - `verification`: 邮箱验证

- **业务功能表**:
  - `file`: 文件管理
  - `payment`: 支付记录
  - `paymentEvent`: 支付事件日志

**特点**:
- 使用 Drizzle 的 `pgTable` 定义表结构
- 完整的外键约束和级联删除
- 自动时间戳字段
- 类型安全的字段定义

### 3. 类型定义层 (`src/server/db/types.ts`)

```typescript
export class DatabaseError extends Error {
```

## Repository 层设计

### Base Repository (`src/server/db/repositories/base-repository.ts`)

提供了标准的 CRUD 操作接口和抽象基类：

```typescript
export interface IBaseRepository<TSelect, TInsert> {
  findById(id: string): Promise<TSelect | undefined>;
  findAll(): Promise<TSelect[]>;
  create(data: TInsert): Promise<TSelect>;
  update(id: string, data: Partial<TInsert>): Promise<TSelect | undefined>;
  delete(id: string): Promise<boolean>;
  exists(id: string): Promise<boolean>;
  count(): Promise<number>;
}
```

**设计优势**:
- 统一的接口规范
- 泛型支持，类型安全
- 标准化的错误处理
- 可扩展的抽象基类

### 具体 Repository 实现

#### 1. File Repository (`src/server/db/repositories/file-repository.ts`)

**核心功能**:
- 文件记录的 CRUD 操作
- 支持分页和搜索
- 用户权限验证
- 关联查询（文件与用户）

**特色方法**:
- `findByUserId()`: 按用户查询文件
- `findAll()`: 管理员查看所有文件（含用户信息）
- `deleteByUserId()`: 用户权限验证删除

#### 2. Payment Repository (`src/server/db/repositories/payment-repository.ts`)

**核心功能**:
- 支付记录管理
- 订阅状态跟踪
- 支付事件日志
- 复杂查询支持

**业务特点**:
- 支持多种支付类型（一次性、订阅）
- 试用期管理
- 取消订阅逻辑

### Repository 层职责

1. **数据访问抽象**: 封装所有数据库操作细节
2. **查询优化**: 提供高效的数据库查询
3. **类型安全**: 利用 Drizzle 的类型推断
4. **错误处理**: 统一的异常处理机制
5. **业务无关**: 专注于数据操作，不包含业务逻辑

## Server Actions 层

### 作用定位

Server Actions 作为 Next.js App Router 的服务端函数，是前端和后端的桥梁：

```typescript
export async function uploadFileAction(formData: FormData): Promise<FileUploadResponse> {
  // 1. 身份验证
  const session = await auth.api.getSession({ headers: await headers() });
  
  // 2. 权限检查
  if (!session?.user) {
    throw new Error(await getErrorMessage('unauthorizedAccess'));
  }
  
  // 3. 调用 Service 层
  const fileInfo = await uploadFile(file, session.user.id);
  
  // 4. 返回前端所需格式
  return { success: true, file: fileInfo };
}
```

**Server Actions 职责**:
1. **请求验证**: 身份认证和权限检查
2. **参数处理**: 解析和验证前端传递的参数
3. **Service 调用**: 调用相应的 Service 层方法
4. **响应格式化**: 将结果格式化为前端需要的格式
5. **错误处理**: 捕获和转换错误信息

## 分层架构优势

### 1. 关注点分离
- **Repository**: 专注数据访问
- **Service**: 专注业务逻辑
- **Actions**: 专注请求处理

### 2. 可测试性
- 每层都可以独立进行单元测试
- 依赖注入便于 Mock 测试
- 清晰的接口定义

### 3. 可维护性
- 代码组织清晰，职责明确
- 修改某一层不影响其他层
- 便于代码复用和扩展

### 4. 类型安全
- 端到端的 TypeScript 类型支持
- Drizzle ORM 提供编译时类型检查
- 减少运行时错误
